# **SPP Distribution System**

# 

# Прототип микросервисного модуля управления иерархическим справочником СПП с историчностью изменений, привязкой отделов и каскадным распределением сумм.

# 

# **Стек технологий**

# 

# \- Backend: Python 3.11, Flask, Flask-SocketIO, SQLAlchemy, Redis, openpyxl

# \- Frontend: Next.js 14, React 18, Zustand, Socket.IO-client, Axios

# \- База данных: PostgreSQL 16 (3NF, SCD Type 2)

# \- Кэш / временное хранилище: Redis 7

# \- Контейнеризация: Docker + Docker Compose

# 

# **Требования**

# 

# \- Docker 24+

# \- Docker Compose v2

# \- Свободные порты: `3000` (фронтенд), `5000` (бэкенд), `5432` (PostgreSQL), `6379` (Redis)

# 

# **Быстрый запуск**

# 

# ```bash

# git clone https://github.com/jdfxhl/testovoe\_zadanie.git

# cd testovoe\_zadanie

# docker-compose up --build

# 

# После старта сервис будет доступен: http://localhost:3000

# 

# **Схема работы «Redis → БД → Real-time»**



1. # Расчёт распределения

# Пользователь выбирает элементы в дереве, задаёт сумму и нажимает «Выполнить».

# Бэкенд (/api/distribution/calculate) вычисляет распределение, сохраняет результат в Redis (ключ distribution:<UUID>) с TTL 24 часа.

# 

# 2\. Сохранение в БД

# По нажатию «Сохранить» бэкенд переносит данные из Redis в таблицу distribution\_results (PostgreSQL, поле distribution\_data типа JSONB).

# После успешной вставки сервер отправляет событие distribution\_saved через WebSocket (Flask-SocketIO) в комнату, соответствующую session\_id пользователя.

# 

# 3\. Real-time синхронизация

# Все вкладки (в рамках одной сессии) подписаны на событие distribution\_saved. При получении события фронтенд автоматически обновляет список сохранённых расчётов без перезагрузки страницы.

# 

# 

# **Функциональность**



1. # Иерархическое дерево СПП (минимум 3 уровня) с чекбоксами и автоматическим выделением всех потомков при выборе родителя.

# 

# 2\.  Исторические срезы: выбор даты в комбобоксе перестраивает дерево на состояние справочника в указанный день (SCD Type 2).

# 

# 3\.  Распределение суммы:

# 

# 3.1  Сумма делится поровну между выбранными элементами.

# 

# 3.2  Каждая ветка рекурсивно распределяет свою долю между активными листьями.

# 

# 3.3  Родительские узлы агрегируют суммы потомков.

# 

# 3.4  Все значения округляются до двух знаков после запятой.

# 

# 4\.  Сохранение результатов: временное хранение в Redis, затем перенос в PostgreSQL (JSONB).

# 

# 5\.  Просмотр сохранённых расчётов и загрузка любого из них обратно в активное дерево.

# 

# 6\.  Экспорт в Excel: плоская таблица с иерархической нумерацией (1, 1.1, 1.2…), суммами и привязкой к отделам.

# 

# 

# **Архитектурные решения**



1. # 3NF и SCD Type 2 для хранения иерархического справочника с историей: каждая модификация элемента создаёт новую запись в spp\_history с valid\_from/valid\_to.

# 

# 2\.  JSONB для результатов распределения – позволяет хранить полный снимок распределённой структуры без приведения к реляционной форме, что упрощает сохранение и загрузку.

# 

# 3\.  Redis как промежуточный кэш – обеспечивает быстрый доступ к результатам расчёта до момента явного сохранения, снижая нагрузку на БД.

# 

# 4\.  WebSocket (Socket.IO) для real-time – клиенты в одной сессии мгновенно получают обновления списка сохранённых расчётов без polling-запросов.

# 

# 5\.  Zustand для управления состоянием фронтенда – лёгкий и производительный стейт-менеджер, идеально подходящий для интерактивных деревьев.

# 

# 6\.  Docker Compose – все сервисы изолированы, легко разворачиваются одной командой.
